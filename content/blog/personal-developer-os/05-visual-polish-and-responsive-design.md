# Phase 5：视觉细化、响应式和交互反馈

做到 Phase 4 之后，网站的核心功能已经差不多可用了。

Main App 能展示内容，Console App 能输入命令，窗口系统能打开、关闭、最小化和最大化，主题、语言、preset 也都能切换。站在工程角度看，继续往博客系统、SEO 或部署推进也说得过去。

但我当时没有急着上线。原因很现实：功能可用不等于适合作为作品展示。

个人网站尤其敏感。访问者不一定会读完整个实现过程，很多判断发生在前几秒。如果界面层级乱、浅色模式粗糙、移动端挤在一起、动效显得廉价，那么再多功能也很难留下好印象。Phase 5 就是专门处理这些“不影响功能，但影响质感”的问题。

## Phase 5.1：先做视觉一致性检查

第一步不是加新样式，而是看哪里不一致。

比如 System Status Bar 的 hover 色一开始有些地方太重，黑白切换比较生硬。AppWindow 最大化后边框层级不够清楚，窗口和背景容易糊在一起。Main App 导航在一些浏览器里滚动条表现不统一。Settings toggle 的 hover 背景也有一点各自为政。

这些问题单独看都不大，但放在一起会让界面显得不够稳。

所以 Phase 5.1 主要做的是“收口”：统一 hover、边框、背景、滚动条和标题栏状态。它没有增加功能，也没有改变窗口行为，只是把已有界面整理得更像同一个系统。

## Phase 5.2：拉开 macos 和 vercel 的差异

这个项目从一开始就有两个视觉 preset：macos 和 vercel。

早期它们已经能切换，但差异还不够明确。macos 应该有玻璃窗口、浮动感和柔和背景；vercel 应该更平、更干净、更像开发者工具。如果两者只是颜色略有不同，那 preset 就没有存在感。

Phase 5.2 里，我把 macos 的窗口背景和内容背景透明度调高，让玻璃感保留的同时提高可读性。卡片背景也变得更实，不再因为过度透明影响文字。阴影做了收敛，不追求很重的悬浮效果，而是让窗口浮起来就够了。

vercel preset 则往另一个方向走：减少玻璃感，边界更清楚，卡片背景更接近实色，窗口阴影很轻。这样同一套组件切换 preset 时，真的能感觉到两种气质。

这里比较重要的一点是没有复制组件。视觉差异仍然通过 style tokens 和 class 映射完成。否则短期看起来方便，后面每改一个内容模块都要同步两份，维护会很痛苦。

## Phase 5.3：light 和 dark 都要能看

很多开发者网站容易把暗色模式做得很用心，亮色模式就像顺手附带的。

我不太想这样。Personal Developer OS 如果是一个长期维护的个人入口，light / dark 都应该站得住。尤其求职或作品展示场景里，访问者的系统主题不一定是暗色，亮色模式不能像半成品。

Phase 5.3 主要修了两个方向。

一是亮色模式下的边界。macos light 里的卡片边框原来太淡，背景又有透明度，导致层级不够清楚。调整后，卡片和嵌套面板的边界更容易看出来。

二是暗色模式下的对比。部分 vercel dark 卡片边框太弱，窗口 chrome 不够利落。调整之后，深色背景里信息层级更清楚，不会只剩一片暗。

这一阶段让我再次确认：主题不是简单换颜色。它影响的是阅读、层级和停留感。

## Phase 5.4：移动端不能只是“能打开”

个人网站一定会有人在手机上看。

早期布局在桌面上还可以，但移动端会暴露很多细节问题：状态栏空间不够、窗口标题可能溢出、Console prompt 太长、桌面图标位置不自然、Hero 标题在窄屏上显得过大。

Phase 5.4 做了几类调整。

状态栏增加了更紧凑的 Portfolio / Console 入口。Console prompt 在移动端从 `visitor@dev-os:~ $` 缩短成 `dev-os:~ $`。AppWindow 标题栏加了 `truncate` 和 `min-w-0`，避免标题把窗口控制区挤坏。Hero 标题在移动端降低字号，不再强行占据太多空间。

这些都不是大改，但移动端体验就是靠这些小修小补撑起来的。我的目标不是让手机端和桌面端完全一样，而是让它在 360px 到 430px 的宽度里仍然像一个完整的 Developer OS。

## Phase 5.5：动效要克制

最后补的是交互反馈。

窗口打开时有轻微 fade-in-scale，按钮点击时有一点 active scale，内容卡片 hover 时轻轻上移，Console 新输出有很短的淡入。它们的存在感都不强，但能让界面从静态截图变成一个可操作的系统。

我刻意没有做夸张动画。

这个网站不是游戏，也不是视觉实验。动画太多会抢内容的注意力，还可能让 OS 概念变成表演。对个人作品集来说，动效最好是让人感觉界面有响应，而不是让人记住动画本身。

同时也加了 `prefers-reduced-motion` 支持。用户系统如果选择减少动态效果，动画就应该停下来。这是一个小细节，但我觉得它属于基本的界面礼貌。

## 视觉阶段真正解决的问题

Phase 5 没有新增业务功能，却明显提高了网站的展示价值。

它解决的不是“能不能用”，而是“别人愿不愿意继续看”。窗口层级更清楚，两个 preset 更有区别，亮色和暗色都能阅读，移动端不再只是勉强打开，交互反馈也不再僵硬。

对我来说，这一步还有一个提醒：AI 可以很快生成界面，但视觉一致性很难靠一次生成解决。它需要反复看、反复收口，尤其要保护已有行为。不能为了一个更漂亮的卡片，把窗口系统、Console 或 preset 切换改坏。

Phase 5 结束后，Personal Developer OS 才更像一个可以公开展示的作品。下一步就可以放心进入内容系统：把博客从几条 mock 数据，变成真正可发布、可索引、可维护的 Markdown Blog。
