# 为什么我要重新做一个个人博客网站

我以前有一个基于 Hexo 的静态博客。

那个博客很简单也很稳定。平时想要记录一些东西只要写一篇Markdown，跑一下生成，推送到线上就结束了。对"把自己的想法放到一个属于自己的博客上"这件事来说，Hexo完全够用。不需要服务端，不需要维护数据库，不需要使用一些公开的平台，主题生态也比较成熟，对个人开发者来说是一种很轻的写作方式。

所以这次重做网站，不是因为Hexo有什么问题。准确说，是我想让个人网站承担的东西变多了。

## 从内容容器到个人作品

以前我把博客当成一个内容容器：文章观点表达出来，页面能访问、不难看，甚至搜索引擎能不能收录都不重要。只要别人问我一个问题恰好我解决过，此时把自己的网站地址给人一丢，觉得就够了，这种感觉很酷，因为7-8年前“会”做这个的很少。

但是后来，特别是今年opc的概念让大家想法慢慢变了。一个个人网站不一定只是文章列表。尤其对开发者来说，网站本身也可以是一个作品——你组织信息的方式、设计的交互、拆分的系统结构、处理工程约束的思路，这些都能从网站里看出来。一个后端开发可以打破自己原有的技术栈，去尝试全栈的开发内容，去尝试自己没有接触到的领域，并且成本可能只是消耗一些token。

从原来Hexo的框架来说，写文章确实好用，但不太适合表达上面这些东西。它更像一个已经定型的框架：我可以选主题、改样式、填内容，但整体体验的骨架基本已经定死了。做普通博客没问题，但如果我希望网站同时当作品集展示、求职入口和技术沉淀，最好能再表现一点我对新鲜事物的探索之心，它就显得有些力不从心。

## Vibe Coding 带来的变化

最近一两年，Vibe Coding、AI 辅助开发、Agent 工具逐渐成熟。它们没有让写代码变成不需要动脑子的事，但确实降低了个人开发者从想法到可运行原型之间的门槛。

以前我会觉得"从零做一个个人网站"性价比不高。现成方案那么多，自己动手意味着要处理设计、前端、内容系统、SEO、部署、维护——如果只是为了写博客，这些投入确实不太划算。

但有 AI 工具帮忙之后，我可以把精力更多放在设计判断、架构拆分和验收标准上，不用在重复的实现细节上花太多时间。这让我重新想了想这件事：如果个人网站本身就是一个长期项目、学习项目和展示项目，那自己做就不只是"重复造轮子"，而是在用真实项目训练和展示完整的工程能力。

这也是我决定重做网站的直接原因。

## 新网站要承担更多角色

这次的新网站不只是一个博客。

它首先还得能写文章。技术学习笔记、AI Agent 实践记录、项目复盘、部署踩坑，这些内容需要有一个稳定的地方放。

但除此之外，我还希望它做几件事：

- 作为求职展示入口，让招聘方快速了解我做过什么、会什么、关注什么方向。
- 作为作品集，展示我怎么把一个想法拆成阶段、模块和可验收的功能。
- 作为 AI 辅助开发的长期记录，保留每个阶段的设计取舍。
- 有一点自己的辨识度，而不是看着像又一个模板站。

这些需求叠在一起之后，一个博客主题就不够用了。我需要的是一个更像产品的个人网站。

## 为什么是 Personal Developer OS

后来的方向慢慢收敛到一个概念：Personal Developer OS。

它不是传统的长页作品集，也不是单纯的文章站，而是一个跑在浏览器里的轻量桌面 OS。首页有 System Status Bar，有 Main App，有 Console App，有 Desktop fallback。用户可以切换 macos / vercel 两种视觉 preset，也可以切换 light / dark 和 zh / en。内容还是个人介绍、技能、项目、博客和联系方式，但它们被组织在一个应用窗口里，不是普通页面的区块。

这个设计不是为了炫技。对我来说，它更像是一个容器——一边展示内容，一边展示我如何设计一个小系统。

这个系列会继续记录从 Hexo 到 Personal Developer OS 的完整过程。下一篇会写为什么我把个人网站设计成 Developer OS，后面还会写 OS Shell 实现、Main App 内容模块、CLI 命令系统、Markdown Blog、SEO、RSS、Docker 部署，以及 v1.0 上线后的复盘。

这不是什么大项目，但也正因为够具体、够小，我希望可以把它认真做完，让它成为一个能持续维护的个人开发者入口。
